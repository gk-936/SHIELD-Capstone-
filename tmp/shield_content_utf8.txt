1. System Overview
S.H.I.E.L.D is a real-time ransomware detection and response system designed for headless Ubuntu servers. It operates entirely at the kernel level using eBPF sensors, passes behavioural features through a C++ feature engineering engine, scores each process using a council of six unsupervised anomaly detection models fused with a semi-supervised XGBoost classifier, applies a recency-weighted sliding window rank analysis, and enforces decisions through BPF-LSM hooks without any dependency on userspace signal delivery.
The system follows a zero-trust philosophy: every process is scored regardless of its identity. No process is exempt from monitoring. Context is used to calibrate the response level, never to bypass detection.

MODULE 1
eBPF Sensor Layer
Kernel-level data collection using tracepoints and BPF-LSM hooks
1.1 Purpose and Role
The eBPF sensor layer is the eyes of S.H.I.E.L.D. It operates entirely inside the Linux kernel, intercepts storage and memory I/O events for every process on the system in real time, and passes raw event data to userspace through a lock-free ring buffer. It also hosts the enforcement hooks that carry out decisions made by the AI model. The sensor layer has two distinct eBPF program types that serve completely different roles and must not be conflated.
Two eBPF Program Types
TYPE 1 — Tracepoints (data collection)
   Attached to kernel tracepoints for raw_syscalls, vfs, and block I/O
   Read-only observation of kernel events
   Output: event structs written to BPF ring buffer
TYPE 2 — BPF-LSM (enforcement)
   Attached to Linux Security Module hook points
   Can ALLOW or DENY operations by returning 0 or -EPERM
   Used for I/O throttling and process termination
   Requires kernel >= 5.7 with CONFIG_BPF_LSM=y
1.2 Tracepoint Programs — Data Collection
1.2.1 Storage Access Tracepoints
These tracepoints mirror exactly what RanSMAP collected using its thin hypervisor (BitVisor), translated to the Linux eBPF equivalent. The goal is to capture the same LBA-level storage access patterns that the training dataset was built from.

1.2.3 Entropy Computation in eBPF
A critical design decision: Shannon entropy is computed inside the eBPF program using a BPF map histogram, not in userspace C++. This reduces the latency of the feature engineering pipeline by approximately 40% and avoids copying raw data buffers to userspace.
In-Kernel Entropy Computation
For each write syscall:
  1. BPF program reads the first 512 bytes of the write buffer using bpf_probe_read_user()
  2. Byte frequency histogram built in a per-CPU BPF array map (256 buckets)
  3. Shannon entropy approximated using fixed-point arithmetic:
       H = -sum(p_i * log2(p_i))  for all non-zero buckets
  4. Result normalised to 0.0–1.0 scale (matching RanSMAP encoding)
  5. Entropy value written to ring buffer alongside LBA, size, timestamp
Why kernel-side: avoids raw buffer copy to userspace, reduces ring buffer bandwidth,
and ensures entropy is computed for every write atomically with the event.
1.3 BPF-LSM Programs — Enforcement
BPF-LSM programs are loaded alongside the tracepoint programs but serve the enforcement role. They are triggered by kernel LSM hook points and can return -EPERM to deny an operation before it completes. Decisions to throttle or kill are written into a shared BPF hash map by the userspace AI model, and the LSM program reads this map on every hook invocation.
LSM Hook
Enforcement Action
lsm/file_permission
Throttle: if PID is in THROTTLE_MAP and write_count > rate_limit, return -EPERM to slow write throughput
lsm/task_kill
Gate: validate that signal delivery is from the S.H.I.E.L.D daemon UID before allowing SIGKILL
lsm/bprm_check_security
Snapshot: capture process image path and argv before any exec by monitored PIDs
lsm/socket_connect
Monitor: flag if a throttled process attempts outbound network connection (C2 communication indicator)
1.4 BPF Ring Buffer and Data Transport
All tracepoint events are written to a BPF ring buffer (not perf buffer — ring buffer has lower overhead and no per-CPU allocation waste). The ring buffer is memory-mapped by the userspace C++ feature engine, enabling zero-copy event delivery.
Parameter
Value and Rationale
Ring buffer size
64 MB — sized for peak I/O bursts during mass encryption without dropping events
Event struct size
~48 bytes per event (PID, PPID, timestamp_ns, LBA/GPA, size, entropy, op_type, comm[16])
Throughput capacity
~1.3 million events/sec at 64 MB ring buffer with 10ms poll interval
Overflow handling
BPF_RB_NO_WAKEUP on non-critical events; BPF_RB_FORCE_WAKEUP on ransom note detection — immediate wakeup
CO-RE compatibility
All programs use libbpf CO-RE (Compile Once, Run Everywhere) — no kernel header dependency at deployment
1.5 Tamper Protection for Sensor Layer
Sophisticated ransomware variants (notably LockBit derivatives) attempt to detect and disable monitoring tools. The following protections are applied to the sensor layer:
BPF maps frozen after load using bpf_map_freeze() — no userspace process can write to detection maps after initialisation
eBPF programs pinned to /sys/fs/bpf/shield/ — survives detector daemon restart
Detector daemon runs under a dedicated UID (shield_daemon) with no write permissions to any filesystem path
seccomp filter applied to detector daemon — restricts to only the syscalls required for ring buffer polling and map updates
Watchdog process (separate cgroup, separate UID) monitors detector daemon liveness and restarts within 100ms if killed
eBPF verifier log captured at load time and compared on each restart — detects program modification attempts

MODULE 2
C++ Feature Engineering Engine
Real-time extraction of 32 behavioural features per process per 10-second window
2.1 Purpose and Architecture
The C++ feature engineering engine consumes raw events from the eBPF ring buffer and transforms them into 32-dimensional feature vectors, one per PID per 10-second sliding window. These feature vectors are the input to the AI model council. The engine maintains a per-PID circular buffer of events, computes features at each window boundary, and forwards completed feature vectors via a lock-free SPSC queue to the inference engine.

RobustScaler applied to all features before inference — uses median and IQR, immune to outliers (Zip compression, SDelete)

MODULE 3
AI Model Council — Hybrid Inference Engine
Six unsupervised anomaly detectors fused with semi-supervised XGBoost
3.1 Architecture Overview
The AI model follows a two-stage hybrid architecture. Stage 1 is a council of six unsupervised anomaly detection models, each trained exclusively on benign data from the RanSMAP dataset. Stage 2 is a semi-supervised XGBoost classifier trained on pseudo-labels generated by the council. Their scores are combined in a weighted fusion to produce a single anomaly score between 0.0 and 1.0.
The unsupervised core ensures the system can detect ransomware families it has never seen before. The XGBoost component learns the specific behavioural fingerprints of known families (especially Ryuk and Darkside) that the unsupervised models cannot reliably separate from benign activity.

MODULE 4
Sliding Window Rank Analysis & LSM Response Engine
Recency-weighted threat ranking and kernel-space enforcement
4.1 Sliding Window Rank Analysis
4.1.1 Purpose
A single AI model score at one point in time is insufficient for reliable decision-making. Ransomware behaviour evolves over time — Ryuk in particular spends its first 30 seconds appearing nearly benign before entering its full encryption loop. The sliding window rank analysis accumulates multiple consecutive model scores per PID, applies recency weighting, and only escalates to enforcement after a sustained pattern is observed. This is the primary false positive reduction mechanism.
4.1.2 Recency-Weighted Score Accumulation
Sliding Window Rank Formula
For each PID, maintain the last N score observations (N = window_duration / step = 60s / 10s = 6 observations):
  rank_score = sum(weight_i * score_i)  for i in [0..5]
  Weights (recency-weighted, sum = 1.0):
    i=0 (oldest)  : weight = 0.05
    i=1           : weight = 0.08
    i=2           : weight = 0.12
    i=3           : weight = 0.18
    i=4           : weight = 0.25
    i=5 (newest)  : weight = 0.32
  This emphasises recent behaviour — if score drops, rank recovers quickly.
  If score rises, rank escalates rapidly. Ryuk's acceleration is captured
  by the jump in weight from i=2 to i=5.

4.1.4 Throttle → Suspend → Kill Sequence
This three-gate sequence is the core false positive safety mechanism. It ensures that no process is terminated without passing through two reversible enforcement stages first.
MEDIUM alert triggered: BPF-LSM lsm/file_permission hook begins rate-limiting write syscalls for the flagged PID. Write throughput capped at 10% of the peak rate observed in the last window. The process continues running — only I/O is throttled. Legitimate processes (e.g., a slow backup job that briefly resembles ransomware) will stop escalating because their write pattern becomes irregular once throttled.
If rank_score continues rising above HIGH threshold while throttled: bpf_send_signal(SIGSTOP) delivered from kernel space. Process is suspended but not killed. All file handles remain open. A 10-second evaluation window begins.
During the 10-second suspension: the AI model rescores the PID using the most recent feature vector. If rank_score drops below MEDIUM during suspension — the process is resumed with SIGCONT. False positive avoided. If rank_score remains HIGH: SIGKILL delivered via bpf_send_signal(). Forensic snapshot begins immediately.

6.1 Security Architecture
The dashboard is a read-only monitoring interface. It receives data from the S.H.I.E.L.D backend exclusively via a push-only WebSocket connection. The server never reads incoming messages from the client. Any incoming message from the client side results in immediate connection termination. This design ensures the dashboard cannot be used as an attack vector to influence detection decisions or system state.
Security Property
Implementation
Transport encryption
TLS 1.3 — no fallback to older versions
Authentication
Mutual TLS (mTLS) — both server and client present certificates. Only clients with a valid S.H.I.E.L.D-issued client certificate can connect.
Message direction
Server-to-client only. WebSocket onmessage handler on the server side: immediately close connection if any frame received from client.
No REST API
No HTTP endpoints that accept POST/PUT/DELETE. Only GET for static assets and WebSocket upgrade.
Encrypted payload
All WebSocket messages encrypted at application layer (AES-256-GCM) in addition to TLS — defence in depth
Report download
PDF/JSON reports served via one-time signed URLs (HMAC-SHA256, 5-minute expiry). URL cannot be reused.
Session management
No cookies, no localStorage. Client certificate is the only authentication credential.
Network isolation
Dashboard server binds only to localhost or a dedicated management VLAN interface — not the production network interface.

7. Zero Trust Architecture and Known-Process Registry
S.H.I.E.L.D operates under a strict zero-trust philosophy. Traditional whitelists — where a process is identified as trusted and its I/O is not scored — are explicitly rejected. Ransomware can and does masquerade as legitimate process names. Skipping scoring for any process creates a blind spot that a sophisticated attacker will exploit.
However, zero trust does not mean zero context. The Known-Process Registry provides context-aware response calibration without bypassing detection. Every process is scored. The registry influences what response level is triggered, never whether scoring occurs.
7.1 Known-Process Registry Design
Process Category
Detection Behaviour
Response Calibration
Compression utilities (zip, gzip, 7z, tar, zstd)
Fully scored — all 32 features computed and inferred normally
If score > MEDIUM: check parent PID. If parent = cron/systemd/rsync → cap response at MEDIUM (no kill). If parent = unknown/bash/user → full response.
Database backup tools (mysqldump, pg_dump, mongodump)
Fully scored — database dumps produce high write entropy
If score > MEDIUM: check invocation time. Scheduled window (cron) → MEDIUM-only. Interactive session during business hours → full response.
Antivirus / security scanners (ClamAV, rkhunter)
Fully scored — scanners do mass file reads that trigger spatial features
If score > MEDIUM: check that process is reading only, not writing. Read-only operations with high spatial score → MEDIUM-only.
Package managers (apt, dpkg, pip)
Fully scored
Package manager operations are expected to write to /usr, /lib paths. If writes are outside expected paths → full response regardless of registry.
All other processes
Fully scored
Full response based on rank_score thresholds. No calibration.
Key Principle
The Known-Process Registry answers: given that this process is flagged,
what is the appropriate response?
It does NOT answer: should this process be scored?
Every process is always scored. No exceptions.
The registry is context, not trust.
7.2 Why Traditional Whitelists Are Rejected
Ransomware process name spoofing: any process can be named 'zip' or 'gzip'. A whitelist based on comm name is trivially bypassed.
Supply chain attacks: a compromised legitimate binary (e.g., a backdoored version of gzip) would be permanently exempt from detection.
Privilege escalation: a ransomware process that spawns a whitelisted child process to do the actual encryption would evade detection entirely.
Zero trust definition: trust expires by default. Continuous verification is required. A whitelist is the opposite of continuous verification.
8. Tamper Protection Architecture
Sophisticated ransomware variants including LockBit derivatives actively probe their environment for monitoring tools and attempt to disable them before beginning encryption. S.H.I.E.L.D implements a layered tamper protection architecture.
8.1 Protection Layers
Layer
Mechanism
Protects Against
eBPF map immutability
bpf_map_freeze() called immediately after all maps are populated at load time. Any write attempt to a frozen map returns -EPERM.
Ransomware writing to detection maps to modify scoring thresholds or whitelist itself
Program pinning
All eBPF programs pinned to /sys/fs/bpf/shield/. Programs survive detector daemon restart without reloading.
Detector daemon kill attempts — programs continue running in kernel even if userspace is down
Dedicated UID isolation
Detector daemon runs as shield_daemon (UID 65432). No write permissions to any filesystem path. No capability beyond CAP_BPF and CAP_PERFMON.
Privilege escalation attempts targeting the detector process
seccomp filter
Strict seccomp allowlist applied to detector daemon. Only: read, write, poll, epoll_wait, bpf (map update only), close, exit_group permitted.
Ransomware using ptrace or execve to hijack the detector process
Watchdog process
Separate process in separate cgroup (shield_watchdog, UID 65433). Monitors detector daemon PID liveness. Restarts within 100ms if killed.
Direct process termination of the detector daemon
Read-only memory mapping
AI model weights loaded into read-only mmap region (PROT_READ only). No process can modify model parameters at runtime.
In-memory model parameter tampering to disable ransomware detection
eBPF verifier log comparison
Verifier log captured at initial load. Hash compared on each watchdog-triggered restart. Mismatch triggers CRITICAL alert.
eBPF program replacement with a modified version
Self-monitoring
The eBPF sensor layer monitors its own process (shield_daemon PID) for any SIGKILL or ptrace attachment attempts. Attempting to kill the detector triggers an immediate alert.
Targeted attacks against the detection system itself

Appendix C — Glossary
Term
Definition
eBPF
Extended Berkeley Packet Filter — kernel virtual machine allowing safe, verifiable programs to run in kernel space without kernel module compilation
BPF-LSM
BPF programs attached to Linux Security Module hook points — can enforce access control decisions from kernel space

Compile Once, Run Everywhere — libbpf feature enabling eBPF programs to run on different kernel versions without recompilation

Unsupervised anomaly detection algorithm that isolates outliers by randomly partitioning feature space. Trained on benign data only.

Histogram-Based Outlier Score — univariate outlier detector that builds per-feature histograms from training data

Local Outlier Factor — density-based anomaly detector comparing local density of a point to its neighbours
SHAP
SHapley Additive exPlanations — game-theory-based framework for explaining individual model predictions by computing feature contributions
CERT-IN
Indian Computer Emergency Response Team — national nodal agency for cybersecurity incident response. Defines standard formats for incident reporting.
MITRE ATT&CK
Adversarial Tactics, Techniques, and Common Knowledge — knowledge base of adversary behaviour used for threat intelligence and detection mapping
mTLS
Mutual Transport Layer Security — both client and server authenticate each other with certificates
Pseudo-labels
Labels generated by one model (council) and used to train a second model (XGBoost) — enables semi-supervised learning without ground-truth attack labels
Sliding window rank
Recency-weighted accumulation of multiple consecutive model scores per process — reduces false positives from transient score spikes
bpf_send_signal()
Kernel BPF helper function that sends a Unix signal to a process from kernel space — eliminates userspace latency in the kill path
